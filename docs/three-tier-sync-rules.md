## 1. 飞书多维表格（团队协作层）

**职责**: 团队成员共享的实时协作记忆

**写入规则**:
- [ ] 新增团队决策、项目里程碑
- [ ] 跨 Agent 协作的重要信息
- [ ] 叶公的指令和反馈
- [ ] 需要所有 Agent 可见的关键数据

**读取规则**:
- [ ] 每日定时读取（Cron: 每天凌晨 3:00）
- [ ] 触发式读取（收到 @提及 或关键词）
- [ ] 跨会话上下文需要时读取

**同步目标**: MEMORY.md（实时或定时）

**字段定义**:
| 字段 | 类型 | 用途 |
|------|------|------|
| 主题 | 文本 | 记录标题 |
| 内容 | 长文本 | 详细内容 |
| 标签 | 多选 | 分类（项目/决策/技术/回忆） |
| 创建时间 | 日期 | 自动生成 |
| 创建者 | 人员 | 记录创建者 |
| 优先级 | 单选 | 高/中/低 |

## 2. 本地 MEMORY.md（核心记忆层）

**职责**: 核心记忆的持久化，AI Agent Community 的长期记忆

**写入规则**:

- [ ] 本地对话的重要决策和关键信息

**读取规则**:
- [ ] 每次对话开始前读取（主会话）
- [ ] 重要操作前检索（memory_search）
- [ ] 需要历史上下文时读取

**同步目标**: GitHub（Git push）和飞书多维表格（选择性）

**结构**:
```markdown

## 关于我
## 关于用户
## 关键项目
## 飞书多维表格
## 记忆文件索引
## ...
```

## 同步策略

### 策略 A: 实时同步（推荐）

```
飞书多维表格 → 实时 → MEMORY.md → 自动提交 → 定时推送 GitHub
```

**触发时机**:
- 飞书多维表格有新记录或修改
- MEMORY.md 有更新

**执行者**: 小诸葛（协调中枢）

**流程**:
1. 检测飞书多维表格变化（Webhook 或定时轮询）
2. 读取新/修改的记录
3. 同步到 MEMORY.md（去重、归类）
4. Git add + commit
5. 定时 Git push（每 30 分钟或每日 3:30）

---

### 策略 B: 定时同步（稳定）

```
每日 3:00: 飞书多维表格 → MEMORY.md
每日 3:30: MEMORY.md → GitHub
```

**优点**: 简单、可控、不频繁
**缺点**: 同步延迟

**Cron 任务**:
```json
{
  "name": "飞书多维表格同步",
  "schedule": {
    "kind": "cron",
    "expr": "0 3 * * *",
    "tz": "Asia/Shanghai"
  },
  "payload": {
    "kind": "systemEvent",
    "text": "触发飞书多维表格同步任务"
  }
}

{
  "name": "GitHub 备份推送",
  "schedule": {
    "kind": "cron",
    "expr": "30 3 * * *",
    "tz": "Asia/Shanghai"
  },
  "payload": {
    "kind": "systemEvent",
    "text": "触发 GitHub 备份推送"
  }
}
```

---

### 策略 C: 混合同步（最佳）

```
飞书 → MEMORY.md: 实时（重要记录）或定时（每日 3:00）
MEMORY.md → GitHub: 每日 3:30
触发式推送: 重要变更立即推送
```

**优点**: 灵活、兼顾实时性和稳定性
**缺点**: 复杂度较高

**优先级判断**:
| 优先级 | 标签 | 同步方式 |
|--------|------|----------|
| 🔴 高 | 决策、里程碑、紧急 | 实时 + 立即推送 |
| 🟡 中 | 项目进展、技术记录 | 定时（每日 3:00） |
| 🟢 低 | 日常对话、临时记录 | 定时（每日 3:00） |

## 冲突解决

### 场景 1: 飞书和 MEMORY.md 同时修改

**规则**: 飞书优先（团队协作层覆盖核心记忆层）

**解决**:
1. 保留飞书的修改
2. MEMORY.md 回退到飞书版本
3. 在 MEMORY.md 中添加冲突标记
4. 人工审查后决定是否调整

---

### 场景 2: GitHub 和本地有冲突

**规则**: 本地优先（本地最新）

**解决**:
1. Git stash 暂存远程变更
2. 继续本地工作
3. 人工审查后决定是否合并远程变更

---

### 场景 3: 多个 Agent 同时写入飞书

**规则**: 时间戳优先，后写覆盖

**解决**:
1. 飞书自动处理并发写入
2. 在记录中添加 `修改者` 字段
3. 重要变更前先读取最新版本

## 记录生命周期

```
创建 → 分类 → 标签 → 优先级 → 同步 → 备份 → 归档
```

**创建**:
- 来源: 对话、系统事件、飞书输入
- 格式: 结构化（主题/内容/标签/时间/创建者）

**分类**:
- 自动: 根据内容和关键词自动分类
- 人工: 手动调整分类

**标签**:
- 固定: 项目/决策/技术/回忆/系统
- 自定义: 可扩展

**优先级**:
- 高: 决策、里程碑、紧急
- 中: 项目进展、技术记录
- 低: 日常对话、临时记录

**同步**:
- 根据优先级选择同步方式（实时/定时）
- 去重和验证

**备份**:
- Git commit（带时间戳和摘要）
- 推送到 GitHub

**归档**:
- 超过 90 天的低优先级记录移至归档
- 归档位置: `memory/archive/YYYY/`

## 监控和告警

### 监控指标

| 指标 | 阈值 | 告警 |
|------|------|------|
| 同步延迟 | > 30 分钟 | 警告 |
| 同步失败率 | > 5% | 告警 |
| 冲突次数 | > 3 次/天 | 告警 |
| 记录数 | < 100 条 | 提示 |
| GitHub 推送失败 | > 3 次/天 | 告警 |

### 告警方式

- 叶公: 飞书消息
- 小诸葛: 日志记录
- 小叶子: 系统通知
